# QA Device Monitor — демо-стенд «ДО» (QR-бронирование с багами)

Это **копия рабочего демо** `qa-device-monitor-demo`, но код фичи **QR-бронирования**
заменён на раннюю версию из `qa-device-monitor-qr-1ver` — с багами.

Пара демо для конференции:
- `qa-device-monitor-demo` — **«ПОСЛЕ»**: фича исправлена, багов нет.
- `qa-device-monitor-demo-qr-1ver` (эта папка) — **«ДО»**: те же баги, что были в первой версии.

Отличается **только один файл**: `js/modules/devices/devices-qr.js` (взят из `qa-device-monitor-qr-1ver`).
Всё остальное (моки, скрытие вкладок/фильтров, фикс аватара, QR-коды) идентично рабочему демо.

## Что показывать на сцене (баги, которые видны вживую)
Сканируйте те же QR из папки `qr-codes/`:

1. **Бронь неисправного устройства** (`DEV-005`)
   В этой версии открывается обычная форма брони — сломанное устройство **можно забронировать**.
   В «ПОСЛЕ» — экран «Устройство неисправно» без формы. *(в коде: «Note: проверка device.isWorking отсутствует»)*
2. **Нет пометки источника брони** — в историю бронирований не пишется `source: 'qr'`.
   *(в коде: «Note: поле source не заполняется»)*
3. **Нет атомарности** — бронирование пишется простым `update()` без транзакции
   (возможна гонка: два одновременных скана забронируют одно устройство).
4. Нет выбора пользователя для общего аккаунта, выбора проектов, смены даты возврата,
   подтверждения получения; нет `min` у даты возврата; нет focus-trap и ARIA-разметки.

## Запуск
```bash
cd qa-device-monitor-demo-qr-1ver
make run        # http://localhost:8080
```

Проверено Playwright (через реальный UI): приложение стартует, каталог и поиск работают,
обычная бронь проходит, и баги воспроизводятся — `DEV-005` бронируется, `source` в брони отсутствует.
