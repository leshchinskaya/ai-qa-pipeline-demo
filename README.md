# Демо QR-бронирования

Учебно-демонстрационный набор: одна и та же фича (**бронирование устройств по QR-коду**)
в двух версиях — с багами и исправленная — плюс QA-артефакты, по которым эти баги находились.
Цель — наглядно показать, что было не так и как стало лучше.

## С чего начать

```bash
# нужен Python 3
make demo-fixed   # исправленная по подходу версия → http://localhost:8082
make demo-buggy   # первоначальная версия генерации кода фичи  → http://localhost:8081
make clean        # освободить порты
make help         # справка
```

Можно запустить **обе сразу** и сравнить «до/после» рядом.

> Демо полностью работает **на моках**.
> Это статическое веб-приложение — серверная часть не нужна.

### Про QR-сканер
Камере браузера нужен `localhost` или `https`, поэтому запускайте через `make` (а не открывая файл).
Если камеры нет — в окне сканера есть **ручной ввод ID устройства**. Готовые QR-коды устройств
лежат в папке [`qr-codes/`](qr-codes/) (откройте `qr-codes/index.html`): можно вывести на экран и сканировать.

## Что в папке

| Папка / файл | Что это |
|---|---|
| `qa-device-monitor-demo/` | Демо **«ПОСЛЕ»** — фича QR-бронирования без багов |
| `qa-device-monitor-demo-qr-1ver/` | Демо **«ДО»** — ранняя версия с багами |
| `qr-codes/` | QR-коды устройств (`DEV-001…005`, `DEV-999`) для сканирования |
| `CL-05_QR_Booking_v1.1.md` | Требования **v1.1** — сводная таблица бизнес-правил BR.05.1–28. По этой версии прогонялся пайплайн (`results/`) и делалось ревью требований (оценка 42%) |
| `CL-05_QR_Booking.md` | Требования **v1.2** — доработанная полная версия: добавлены разделы про инициализацию экрана, формирование, лоадеры, поля ввода, дизайн и пр. (то, чего не хватало v1.1 по итогам ревью) |
| `results/` | QA-артефакты — **выход инструментов** `qa-autogenerator-manual-tests` и `e2e-test-framework`: требования → ручные тесты → автотесты + матрица трассируемости (см. ниже) |
| `results-terminal/` | Логи пошагового прогона QA-пайплайна — пример работы тех же инструментов |
| `Makefile` | Команды запуска демо (`demo-fixed` / `demo-buggy`); цели `stepN` — справочные, требуют самих фреймворков |

Две демо отличаются **кодом фичи** — `js/modules/devices/devices-qr.js`
(в «ДО» — ранняя версия с багами, в «ПОСЛЕ» — исправленная). Остальной код приложения идентичен
(различается лишь README внутри каждой папки).

## Связь артефактов

```
CL-05_QR_Booking_v1.1.md  →   [инструменты]    →   results/        →   два демо-приложения
   (требования v1.1)        qa-autogenerator +     (ручные + авто-     (видно баги и фиксы)
                            e2e-test-framework      тесты, матрица)
```

> По итогам ревью требований (оценка 42%) ТЗ доработали до **v1.2** (`CL-05_QR_Booking.md`) —
> добавили недостающие разделы. То есть «до/после» есть и у требований, и у приложения.

## Чем сгенерированы артефакты (инструменты)

Содержимое `results/` и логи в `results-terminal/` — это **выход фреймворков**.
Сами фреймворки в этот набор **не входят** (это самостоятельные большие проекты):

- **`qa-autogenerator-manual-tests`** — фреймворк, который на вход берёт требования (ТЗ v1.1),
  делает их ревью и **генерирует ручные тест-кейсы** (компонентные и сценарные).
  Его выход здесь: `results/review_requirements/`, `results/manual_tests/`.
- **`e2e-test-framework`** — фреймворк, который из ручных тест-кейсов
  **генерирует автотесты (e2e, Playwright/TypeScript)** и отчёты о прогоне.
  Его выход здесь: `results/autotests/` (в т.ч. сами спеки в `results/autotests/e2e/qr-booking/`
  и матрица трассируемости).

Пайплайн их запуска (для справки) описан целями в `Makefile`,
но эти цели требуют установленных фреймворков и в публичном наборе не запускаются —
рабочие команды здесь только `make demo-fixed` / `make demo-buggy`.

## Какие баги показывает пара «ДО/ПОСЛЕ»

Примеры (полный список — в коде и QA-артефактах):

- **Бронь неисправного устройства** — в «ДО» сломанный девайс можно забронировать; в «ПОСЛЕ» — экран «Устройство неисправно». *(скан QR `DEV-005`)*
- **Дата возврата в прошлом** — в «ДО» нет ограничения даты; в «ПОСЛЕ» нельзя выбрать прошлую.
- **Нет пометки `source:'qr'`** — в «ДО» QR-брони неотличимы от ручных в истории.
- **Нет атомарной транзакции** — в «ДО» два одновременных скана забронируют одно устройство.
- **Доступность (a11y)**, выбор проектов, выбор пользователя для общего аккаунта, подсказка при долгом скане и др.

## Требования

- **Python 3** (для локального сервера демо).
- Современный браузер; для камеры сканера — запуск по `localhost`/`https`.
