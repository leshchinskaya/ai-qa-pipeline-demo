## 1. Общая оценка полноты ТЗ

**Оценка: 42% — Неудовлетворительно**

ТЗ содержит хорошо проработанные бизнес-требования с критериями приёмки (28 пунктов BR.05.1–BR.05.28), покрывает основные сценарии и крайние случаи. Однако документ представляет собой только таблицу бизнес-требований — **структурные разделы ТЗ полностью отсутствуют**: нет вводной части, технической части, описания экранов, маппинга методов API, визуальных материалов, описания состояний загрузки и полей ввода. Это делает документ непригодным для начала разработки без существенных уточнений.

## 2. Критические замечания

### К.1. Полное отсутствие Swagger-спецификации

**Проблема:** Swagger-спецификация пуста — `swagger.yaml` не содержит никаких данных.  
**Требования:** requirements.md  
**Раздел:** "Swagger-спецификация (опционально)"  
**Строки:** BR.05.4, BR.05.10, BR.05.16, BR.05.18, BR.05.22 — все требования, ссылающиеся на поля Firestore  
**Цитата:** `"Запись bookings содержит deviceId, userId, action = take, корректный bookingType и поле source = \"qr\""` (BR.05.10)  
**Влияние:** В ТЗ активно используются поля Firestore (`status`, `bookingType`, `currentUserId`, `bookedFor`, `pendingReceipt`, `bookings`, `source`, `bookedBy`, `bookedByName` и др.) без формального описания моделей данных. Невозможно валидировать API-контракты. Фронтенд-разработчики не имеют источника истины по структуре документов.  
**Рекомендация:** Добавить swagger/OpenAPI-описание или отдельный раздел «Модели данных» с описанием коллекций Firestore. Минимально необходимые модели:
- `Device` (поля: `deviceId`, `status`, `bookingType`, `currentUserId`, `currentUserName`, `currentUserPhoto`, `bookedAt`, `bookedUntil`, `pendingReceipt`, `bookedFor`, `externalDepartment`, `externalComment`, `isWorking`, `currentProjectIds[]`)
- `BookingRecord` (поля: `deviceId`, `userId`, `action`, `bookingType`, `source`, `createdAt`, `bookedBy`, `bookedByName`)

**Приоритет:** Критический

**Проблема:** В ТЗ нигде не перечислены конкретные методы/запросы, используемые в функциональности. Все технические детали рассыпаны по критериям приёмки.

**Влияние:** Разработчики не знают: какие Firestore-коллекции читаются/пишутся, в каком порядке, какие индексы нужны, как устроен security rules, какие методы вызываются при загрузке экрана, а какие — по действию пользователя.  
**Рекомендация:** Добавить раздел «Список методов» с таблицей вида:

|---|----------------|-----|----------------|-----------------|----------------|-----------------|
| 1 | Получить устройство по deviceId | Firestore.get | `devices/{deviceId}` | После сканирования QR | `deviceId: string` | `Device` |
| 2 | Создать бронь (транзакция) | Firestore.transaction | `devices/{deviceId}`, `bookings` | По нажатию «Подтвердить» | `BookingRecord` | `BookingRecord` |
| 3 | Подтвердить получение | Firestore.update | `devices/{deviceId}`, `bookings` | QR при pendingReceipt | `deviceId`, `userId` | обновлённый `Device` |

### К.4. Отсутствие последовательности формирования экрана

**Проблема:** Нет раздела «Последовательность формирования экрана» — неизвестно, что загружается и в каком порядке.  
**Требования:** requirements.md  
**Раздел:** Отсутствует  
**Влияние:** Разработчик не знает: что показывается пока камера инициализируется, в каком порядке выполняются проверки (авторизация → доступ к камере → сканирование → проверка deviceId → проверка статуса → показ шторки), какие данные prefetch'ятся.  
**Рекомендация:** Добавить нумерованный список последовательности:
1. Проверка авторизации → если нет → OAuth redirect (BR.05.8)
2. Запрос разрешения камеры → если нет → fallback (BR.05.23)
3. Инициализация видеопотока → ожидание сканирования + таймер 10 сек (BR.05.24)
4. Распознавание QR → извлечение deviceId
5. Firestore.get(`devices/{deviceId}`) → показ превью карточки (BR.05.3)
6. Определение состояния устройства → ветвление flow
7. Показ соответствующей шторки → действие → транзакция → результат (BR.05.20)

**Приоритет:** Критический

### К.5. Отсутствие описания состояний загрузки (лоадеров)

**Проблема:** Нигде не указаны типы лоадеров и поведение UI во время выполнения асинхронных операций.  
**Требования:** requirements.md  
**Раздел:** Отсутствует; BR.05.20 упоминает только результат успешной операции  
**Строки:** BR.05.20, BR.05.21  
**Цитата:** `"Toast с текстом результата (бронь / возврат / подтверждение получения); сканер закрывается"` (BR.05.20) — состояние во время операции не описано  
**Влияние:** Неясно, что показывается пока выполняется Firestore-транзакция: spinner на кнопке «Подтвердить», полноэкранный лоадер, блокировка интерфейса. Без этого возможны двойные нажатия и создание дублей, несмотря на BR.05.9.  
**Рекомендация:** Для каждой async-операции явно указать тип лоадера:
- Загрузка устройства по deviceId: скелетон карточки в шторке

- Инициализация камеры: полноэкранный лоадер с текстом «Открываем камеру...»

**Приоритет:** Критический

## 4. Рекомендации

### Р.1. Добавить раздел «Бизнес-цель и юзкейсы»

Текущее описание кластера в одном предложении недостаточно. Рекомендуется добавить:
- Бизнес-цель: `"Ускорить процесс бронирования с 3-4 тапов до 1-2 тапов за счёт автоматического определения устройства по QR-коду"`
- Юзкейсы (User Stories):
  - Как сотрудник, хочу быстро забронировать устройство сканированием QR, чтобы не искать его в каталоге
  - Как сотрудник, хочу подтвердить получение переданного мне устройства сканированием QR
  - Как администратор, хочу видеть статистику QR-сканирований для оценки использования фичи

### Р.2. Добавить декомпозицию на компоненты

Разбить фичу на компоненты:
1. **QR Scanner Screen** — полноэкранный экран с видеопотоком
2. **QR Scanner Toolbar** — переключение камер, кнопка загрузки фото, таймер 10 сек
3. **Device Preview Card** — превью устройства до подтверждения (BR.05.3)
4. **Booking Bottom Sheet** — шторка брони (available state)
5. **Device Status Bottom Sheet** — шторка для booked/external/not_working
6. **Own Device Actions Sheet** — шторка для return/date_change (BR.05.19)
7. **Manual DeviceId Input Sheet** — поле ввода ID вручную (BR.05.24)

### Р.3. Уточнить back navigation

Не описано:
- Как закрывается экран сканера (кнопка «×» / «назад»)?
- Что происходит с состоянием приложения после закрытия сканера
- Куда возвращается пользователь (на вкладку «Устройства» или на предыдущий экран?)
- Сохраняется ли состояние сканера при сворачивании PWA (для возврата через OAuth — BR.05.8)

### Р.4. Описать fallback при ошибке переключения камер

BR.05.11 упоминает переключение между камерами, но не описывает поведение при ошибке переключения. Что происходит, если выбранная камера недоступна (занята другим приложением)? Рекомендуется добавить: `"При ошибке переключения — показать toast 'Не удалось переключить камеру' и оставить текущую активной."`
