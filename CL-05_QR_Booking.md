# Бизнес-требования: QR-бронирование

**UID кластера:** CL-05  
**Название кластера:** QR-бронирование  
**Описание:** Требования к мгновенному бронированию устройства по сканированию его QR-кода камерой телефона или ноутбука. Бронь оформляется на текущего авторизованного пользователя без ручного выбора устройства из каталога. Покрываются выбор типа брони (office/home), проектный контекст, специальные состояния устройства (`pendingReceipt`, `external`, `isWorking = false`), shared-аккаунты, обработка ошибок камеры/Firestore, гонки и fallback (загрузка картинки, ручной ввод).

> Стек приложения — Web + Firebase (PWA, Firestore, Firebase Auth, Firebase Analytics). Отдельных backend-API/Swagger в фиче нет: чтения и записи идут напрямую через Firestore SDK с клиента; security rules защищают данные на уровне Firestore. По этой причине разделы «Список методов API» / «Swagger» в ТЗ отсутствуют намеренно — там, где нужны технические детали, описаны Firestore-операции и затрагиваемые коллекции/поля прямо в критериях приёмки.

---

## 1. Бизнес-цель и юзкейсы

### 1.1 Бизнес-цель
Сократить путь оформления брони с 3–4 действий (открыть каталог → найти устройство → открыть карточку → подтвердить бронь) до 1–2 действий (сканировать QR → подтвердить), а также упростить подтверждение получения переданных устройств и возврат «своих» устройств без поиска в каталоге.

### 1.2 Юзкейсы (User Stories)
- **US-05.1.** Как сотрудник, хочу быстро забронировать устройство сканированием QR, чтобы не искать его в каталоге.
- **US-05.2.** Как сотрудник, хочу подтвердить получение устройства, которое мне передали (`pendingReceipt`), сканированием QR.
- **US-05.3.** Как сотрудник, хочу одним сканированием своего занятого устройства открыть действия «Вернуть» / «Изменить дату возврата».
- **US-05.4.** Как сотрудник, хочу при сканировании выбрать пользователя, на которого оформляется бронь.
- **US-05.5.** Как админ, хочу видеть в аналитике результаты QR-сканирований (успех, занято, не найдено, ошибка), чтобы оценить использование фичи.
- **US-05.6.** Как сотрудник без рабочей камеры, хочу ввести `deviceId` вручную или загрузить картинку с QR и пройти тот же flow, что и при успешном сканировании.

## 2. Точки инициализации экрана сканера

| № | Точка входа | Контекст | Передаваемые данные | Поведение после закрытия |
|---|-------------|----------|---------------------|--------------------------|
| 1 | Кнопка «Сканировать QR» на вкладке «Устройства» | Авторизованный пользователь | — | Возврат на вкладку «Устройства» |
| 2 | Deep link `/#scan` | Любая авторизованная сессия | Нет preloaded deviceId | Возврат на вкладку «Устройства» |
| 3 | Deep link `/#device/{deviceId}` (QR в виде URL) | Любая сессия (если не авторизован — OAuth) | `deviceId` из URL → сразу шаг превью устройства (минуя камеру) | Возврат на вкладку «Устройства» |
| 4 | Возврат после Google OAuth (BR.05.8) | Изначально неавторизованная сессия | `deviceId`, сохранённый в `sessionStorage` (если был распознан до редиректа) | Открывается соответствующая шторка flow |
| 5 | Push/in-app уведомление о передаче устройства (`pendingReceipt`) | Авторизованная сессия | `deviceId` получателя | После подтверждения — toast и возврат к каталогу |

На неавторизованной сессии вход в сканер недоступен напрямую — любая попытка инициирует Google OAuth (BR.05.8).

## 3. Последовательность формирования экрана

Порядок действий при открытии сканера (применимо к точкам входа 1, 2, 4 из раздела 2; для точки 3 шаги 2–4 пропускаются — deviceId уже известен):

1. **Проверка авторизации.** Если `Auth.currentUser === null` — редирект на Google OAuth, deviceId (если есть) сохраняется в `sessionStorage` (BR.05.8).
2. **Запрос разрешения камеры.** `navigator.mediaDevices.getUserMedia({ video: { facingMode: 'environment' } })`. При отказе — переход к fallback (BR.05.23).
3. **Инициализация видеопотока.** Полноэкранный лоадер «Открываем камеру…», пока поток не готов. Параллельно запускается таймер 10 сек (BR.05.24).
4. **Распознавание QR.** Извлечение `deviceId` или URL-вида `/#device/{deviceId}`.
5. **Чтение устройства из Firestore.** `devices/{deviceId}` (одиночный `get`, не подписка). Во время чтения — скелетон карточки. Если документа нет — ошибка «Устройство не найдено» (BR.05.7).
6. **Определение состояния и ветвление flow:**
   - `status === 'available'` → шторка брони с предзаполненным `office` (BR.05.4).
   - `status === 'booked'` и `currentUserId === Auth.currentUser.uid` → экран действий «Вернуть / Изменить дату» (BR.05.19).
   - `status === 'booked'` и `pendingReceipt === true` и `bookedFor === Auth.currentUser.uid` → flow подтверждения получения (BR.05.16).
   - `status === 'booked'` другим пользователем → карточка занятости (BR.05.5).
   - `status === 'external'` → информационная карточка (BR.05.6).
   - `isWorking === false` → предупреждение «Устройство неисправно» (BR.05.17).
   - Shared-аккаунт `qadept@test.dev` → шторка с обязательным выбором пользователя (BR.05.18).
7. **Действие пользователя → Firestore transaction → результат.** Во время транзакции — spinner на кнопке «Подтвердить», блокировка повторного нажатия (см. раздел 5). По результату — toast и закрытие сканера (BR.05.20) или баннер с retry (BR.05.21).

## 5. Состояния загрузки (лоадеры) и блокировки UI

| Операция | Тип лоадера | Блокировка UI | Цель |
|----------|-------------|----------------|------|
| Инициализация камеры | Полноэкранный лоадер «Открываем камеру…» внутри сканера | Toolbar (смена камеры) disabled до готовности | Не дать пользователю взаимодействовать до готовности потока |
| Распознавание QR | Подсказка «Наведите камеру на QR-код устройства» + таймер 10 сек | — | Информировать о текущем шаге; через 10 сек — подсказка с альтернативой (BR.05.24) |
| Чтение `devices/{deviceId}` после распознавания | Скелетон в Device Preview Card / шторке | Кнопка «Подтвердить» скрыта/disabled | Избежать показа пустых полей |
| Firestore transaction (создание брони / возврат / `receipt_confirmed` / `date_change`) | Spinner на кнопке «Подтвердить» (или «Вернуть» / «Подтвердить получение») | Все кнопки шторки disabled до завершения | Исключить дубли (двойной сабмит) и нежелательные действия |
| Поиск пользователя на shared-аккаунте (C8) | Inline-спиннер справа от поля поиска | Кнопка «Подтвердить» disabled, пока пользователь не выбран | Прозрачно показать загрузку результатов |
| Загрузка картинки с QR (fallback) | Спиннер на кнопке «Загрузить картинку» | Кнопка disabled до завершения декодирования | Избежать повторных нажатий |
| Восстановление после OAuth-редиректа (BR.05.8) | Полноэкранный лоадер «Возвращаемся к сканеру…» | Сканер скрыт до восстановления состояния | Передать пользователю контекст возврата |

Общее правило: на время асинхронной Firestore-операции кнопка-инициатор переходит в состояние `loading + disabled`; повторное нажатие невозможно до завершения.

## 7. Работа со шторкой и навигация

### 7.1 Закрытие шторки
- **Тап вне области шторки (overlay)** → закрытие, введённые данные сбрасываются.
- **Свайп вниз (mobile)** → закрытие.
- **Физическая кнопка «Назад» (Android PWA)** → закрытие шторки (а не выход из приложения).
- **`Esc` (web/desktop)** → закрывает шторку и сканер (расширение BR.05.27).
- Если пользователь изменил тип брони, выбрал проект или ввёл дату — перед закрытием показывается диалог «Отменить заполнение? Введённые данные не сохранятся» с кнопками «Отменить» / «Закрыть».

### 7.2 Закрытие сканера
- Кнопка «×» в правом верхнем углу.
- `Esc` (web/desktop) — закрывает сканер целиком.
- При закрытии:
  - Видеопоток освобождается (`stream.getTracks().forEach(t => t.stop())`).
  - `sessionStorage` ключи, относящиеся к QR-flow (deviceId на возврат после OAuth), очищаются, если flow завершён успешно или отменён пользователем.

### 7.3 Back navigation
- После успешной операции (бронь / возврат / `receipt_confirmed` / `date_change`) — toast и возврат на вкладку, с которой был открыт сканер (по умолчанию — «Устройства»).
- После отмены (закрытие сканера или шторки) — возврат на ту же вкладку без изменений каталога.
- После OAuth-редиректа (BR.05.8) — пользователь оказывается на экране сканера; при наличии сохранённого `deviceId` — сразу на превью устройства (раздел 3, шаг 6).

## 8. Обновление данных и работа с каталогом

- Каталог устройств уже использует Firestore real-time подписку (`onSnapshot` на коллекцию `devices`). Любое изменение, сделанное QR-операцией, отразится в списке автоматически — повторный запрос не нужен.
- Карточка устройства внутри QR-flow (Device Preview Card, шторки) использует одиночное чтение (`get`) на момент распознавания; статус повторно проверяется внутри транзакции (BR.05.22) на момент записи.
- Pull-to-refresh (PTR) на вкладке «Устройства» остаётся доступным как ручной триггер ресинхронизации (например, если listener был отключён в фоне PWA).

## 9. Аналитика и наблюдаемость

- События отправляются через **Firebase Analytics** (`firebase.analytics().logEvent(...)`). В offline-режиме события буферизуются Firebase SDK и доставляются при восстановлении сети.
- Событие `qr_scan` (BR.05.28) содержит поля:
  - `result`: `success` / `not_found` / `already_booked_other` / `already_own` / `external` / `not_working` / `error`
  - `source`: `camera` / `upload` / `manual`
  - `device_form`: `mobile` / `desktop`
  - `booking_type`: `office` / `home` / `null`
- Дополнительно при ошибках Firestore (BR.05.21) — событие `qr_scan_error` с полем `error_code` (например, `permission-denied`, `unavailable`, `aborted`).
- Агрегаты доступны в разделе «Активность» админки через подписку на BigQuery-экспорт Firebase Analytics (как и для других событий приложения).

## 10. Бизнес-требования кластера

| ID | Требование | Критерии приемки |
|----|------------|------------------|
| BR.05.1 | Каждое физическое устройство имеет уникальный QR-код, привязанный к его deviceId | В карточке устройства и в админке доступен QR; полезная нагрузка QR содержит только deviceId или короткую ссылку вида `/#device/{deviceId}` |
| BR.05.2 | Авторизованный пользователь может открыть встроенный сканер QR из приложения | На вкладке «Устройства» доступна кнопка «Сканировать QR»; на неавторизованной сессии кнопка скрыта |
| BR.05.3 | Сканер запрашивает доступ к камере и распознаёт QR-код устройства | При первом запуске запрашивается разрешение `getUserMedia`; при распознавании показывается превью карточки устройства до подтверждения |
| BR.05.4 | Если устройство в статусе `available` — после сканирования открывается шторка брони с предзаполненным быстрым вариантом «Использую (office)» | Шторка показывает тип брони `office`, имя пользователя, кнопку «Подтвердить»; при подтверждении — статус `booked`, `bookingType = office`, `currentUserId = Auth.currentUser.uid`, заполнены `currentUserName`, `currentUserPhoto`, `bookedAt`; в `bookings` создаётся запись с `action = take` |
| BR.05.5 | Если устройство уже `booked` другим пользователем — бронь не оформляется, показывается карточка занятости | Бронь не создаётся; отображается `currentUserName` и `bookedUntil` (если задан); кнопка подтверждения брони неактивна с подсказкой «Устройство уже занято» |
| BR.05.6 | Если устройство в статусе `external` — кнопки бронирования скрыты, показывается комментарий | Видны `externalDepartment` и `externalComment`; действия по брони недоступны |
| BR.05.7 | Сканер не должен оформлять бронь, если QR не соответствует ни одному устройству в Firestore | Показывается ошибка «Устройство не найдено», запись в `bookings` не создаётся, статус устройств не меняется |
| BR.05.8 | На неавторизованной сессии открытие сканера перенаправляет на Google OAuth и после входа возвращает к сканеру | Перед редиректом распознанный (или переданный через deep link) `deviceId` сохраняется в `sessionStorage` (ключ `qrPendingDeviceId`); после успешного входа пользователь оказывается на экране сканера; если сохранённый `deviceId` есть — открывается соответствующая шторка минуя камеру |
| BR.05.9 | Повторное сканирование устройства, уже занятого текущим пользователем, не создаёт дубликата брони и переводит пользователя к доступным действиям | При `currentUserId === Auth.currentUser.uid` новая запись `take` в `bookings` не появляется; UI-поведение полностью описано в BR.05.19 (единственный источник истины для экрана «своё устройство») |
| BR.05.10 | Бронирования по QR фиксируются в истории идентично ручным, с пометкой источника | Запись `bookings` содержит `deviceId`, `userId`, `action = take` (или `receipt_confirmed`, `return`, `date_change` в соответствующих сценариях), корректный `bookingType` и поле `source = "qr"` |
| BR.05.11 | Сканер работает на мобильной камере (PWA) и на веб-камере ноутбука с поддержкой переключения между фронтальной и тыловой камерой | Используются доступные `MediaDeviceInfo`; пользователь может выбрать камеру; при ошибке переключения — toast «Не удалось переключить камеру», текущая камера остаётся активной; при отсутствии камеры доступен фолбэк «Загрузить картинку с QR» |
| BR.05.12 | QR-код не содержит чувствительных данных | В полезной нагрузке нет email, токенов, профиля и проекта; только идентификатор устройства |
| BR.05.13 | Все действия по QR-брони отображаются на вкладке «История» наравне с ручными бронями | В timeline появляется запись с актуальным `action`, корректными `deviceName`, `userName` и временем `createdAt` |
| BR.05.14 | В шторке после сканирования пользователь может переключить тип брони на `home` с датой возврата | Доступен переключатель `office / home`; при выборе `home` появляется обязательное поле `bookedUntil` (date picker, мин. = сегодня, макс. = +90 дней); при подтверждении — `bookingType = home`, `bookedUntil` заполнен, поведение совпадает с BR.02.2 |
| BR.05.15 | В шторке можно выбрать один или несколько проектов, как в ручном бронировании | Поле выбора проектов идентично BR.02.2; сохраняются `currentProjectIds[]`, `currentProjectCodes[]`, `currentProjectNames[]` и `currentProjectId/Code/Name` для основного |
| BR.05.16 | Сканирование устройства, переданного на текущего пользователя через `pendingReceipt`, подтверждает получение и не создаёт новую бронь | При `pendingReceipt = true` и `bookedFor === Auth.currentUser.uid` — `pendingReceipt` сбрасывается, `bookedAt` не перетирается, в `bookings` пишется `action = receipt_confirmed` |
| BR.05.17 | Сканирование устройства с `isWorking = false` показывает предупреждение и запрещает бронь | Отображается баннер «Устройство неисправно»; кнопка «Подтвердить» отключена; запись в `bookings` и изменение `status` не выполняются; для админа доступна кнопка «Открыть карточку устройства» |
| BR.05.18 | На shared-аккаунте (`qadept@test.dev`) после сканирования предлагается выбрать пользователя, на которого оформляется бронь | Шторка показывает поиск по пользователям из коллекции `users` (отображаемое имя + email; минимум 2 символа, debounce 300 мс, пагинация по 20, сортировка по `displayName`; empty state «По запросу `…` никого не найдено»); поле обязательно; при подтверждении `currentUserId` = выбранный пользователь, `bookedBy` = shared-аккаунт, `bookedByName` заполнен. Полная логика идентична фиче `bookForOthers` |
| BR.05.19 | Сканирование уже занятого текущим пользователем устройства открывает экран с действиями «Вернуть» и «Изменить дату возврата» | При `currentUserId === Auth.currentUser.uid` — отображается карточка устройства с активными `return` и (для `home`) `date_change`; «Вернуть» вызывает существующий return-flow (BR.02.3) и пишет `action = return` в `bookings` с `source = "qr"`; «Изменить дату» — flow из BR.02.4 с `action = date_change` |
| BR.05.20 | После успешной QR-операции пользователь видит подтверждение и возвращается к каталогу или карточке устройства | Toast с текстом результата (бронь / возврат / подтверждение получения); сканер закрывается; каталог обновляется в реальном времени через `onSnapshot` (раздел 8); ошибки операции не оставляют сканер открытым в неопределённом состоянии |
| BR.05.21 | При ошибке записи (сеть, права, конфликт версий) показывается понятное сообщение и кнопка «Повторить» | При неудачной операции с Firestore — баннер с описанием причины + `retry`; статус устройства не меняется до успешного подтверждения; пользователь не видит «успех» при фактической ошибке; событие `qr_scan_error` с `error_code` отправляется в Firebase Analytics (раздел 9) |
| BR.05.22 | Создание брони атомарно: при гонке двух одновременных сканирований одного устройства бронь получает только один пользователь | Запись брони идёт через Firestore transaction (`runTransaction`) с повторной проверкой `status === 'available'` внутри транзакции; проигравший видит экран занятости (BR.05.5) и понятное сообщение «Устройство только что забронировали» |
| BR.05.23 | При отказе в доступе к камере или её отзыве пользователю доступен альтернативный путь | Показывается подсказка «Доступ к камере запрещён»; доступны кнопки `Загрузить картинку с QR` и `Ввести ID устройства вручную`; после исправления разрешения возможен retry без перезагрузки |
| BR.05.24 | Если QR не распознаётся в течение 10 секунд — показывается подсказка с альтернативой | Подсказка «Не получается распознать? Попробуйте сменить ракурс или ввести ID вручную»; ручной ввод `deviceId` приводит к тому же flow, что и успешное сканирование |
| BR.05.25 | При смене владельца устройства через QR (бронь у другого человека, возврат) предыдущему пользователю отправляется in-app уведомление | На смену `currentUserId` или возврат — push/чат-нотификация прошлому владельцу с `deviceName` и именем нового владельца; admin видит событие в активности |
| BR.05.26 | В offline-режиме (PWA без сети) сканер показывает предупреждение и блокирует создание брони | При `navigator.onLine === false` — баннер «Нет сети, бронирование недоступно»; запись в Firestore не пробуется до восстановления связи; при восстановлении пользователь может вручную повторить операцию |
| BR.05.27 | Экран сканера и шторка брони доступны для клавиатурного управления и screen reader | Все интерактивные элементы имеют `aria-label`; фокус-trap внутри модалки; `Esc` закрывает сканер и активную шторку (раздел 7.1); ручной ввод `deviceId` (BR.05.24) служит альтернативой для пользователей без камеры |
| BR.05.28 | Каждая QR-операция логируется в аналитику с метаданными для оценки использования фичи | Событие `qr_scan` (Firebase Analytics, см. раздел 9) с полями: `result` (`success` / `not_found` / `already_booked_other` / `already_own` / `external` / `not_working` / `error`), `source` (`camera` / `upload` / `manual`), `device_form` (`mobile` / `desktop`), `booking_type` (`office` / `home` / `null`); агрегируется в админке |
